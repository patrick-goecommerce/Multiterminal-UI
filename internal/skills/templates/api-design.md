You are designing or working on a REST API. Follow these principles:

- **Resource-oriented URLs:** use plural nouns for collections (`/users`, `/orders`). Nest for clear ownership (`/users/{id}/orders`). Limit nesting to 2 levels — flatten with filters beyond that (`/orders?user_id=123`). Never use verbs in URLs.
- **HTTP methods:** `GET` for reads (safe, idempotent), `POST` for creation, `PUT` for full replacement, `PATCH` for partial update, `DELETE` for removal. `POST` is not idempotent — use `PUT` or an idempotency key for operations that must not duplicate.
- **Status codes:** `200` success, `201` created (with `Location` header), `204` no content (DELETE), `400` bad request, `401` unauthenticated, `403` forbidden, `404` not found, `409` conflict, `422` validation error, `429` rate limited, `500` server error. Never return `200` with an error body.
- **Error responses:** use a consistent format: `{ "error": { "code": "VALIDATION_FAILED", "message": "Human-readable message", "details": [...] } }`. Include field-level errors in `details` for validation failures. Use RFC 7807 Problem Details for complex APIs.
- **Pagination:** use cursor-based pagination for real-time data (`?cursor=abc&limit=20`). Use offset pagination (`?page=2&per_page=20`) only for static datasets. Always return pagination metadata: `next_cursor`, `has_more`, or `total_count`.
- **Filtering and sorting:** use query parameters for filtering (`?status=active&created_after=2024-01-01`). Use `sort` parameter with field and direction (`?sort=-created_at,name`). Document all supported filter fields and operators.
- **Versioning:** version the API in the URL path (`/api/v1/users`) for simplicity and cacheability. Increment the major version only for breaking changes. Support the previous version for at least 6 months after deprecation.
- **Request/response format:** use JSON (`Content-Type: application/json`). Use camelCase for JSON field names. Wrap single-resource responses in the resource itself (not an envelope). Wrap collection responses with metadata: `{ "data": [...], "pagination": {...} }`.
- **OpenAPI specification:** maintain an OpenAPI 3.x spec as the source of truth. Document all endpoints, request/response schemas, error codes, and auth requirements. Generate client SDKs and server stubs from the spec when possible.
- **Authentication:** use Bearer tokens (JWT or opaque) in the `Authorization` header. Never pass tokens in URLs. Use short-lived access tokens with refresh tokens. Return `401` for missing/invalid tokens, `403` for insufficient permissions.
- **Rate limiting:** return `429 Too Many Requests` with `Retry-After` header. Include rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`). Apply different limits per endpoint sensitivity (auth endpoints get stricter limits).
- **Idempotency:** support `Idempotency-Key` header for POST operations that create resources or trigger side effects. Store the response and replay it for duplicate keys. Key expiration should be documented (typically 24-48 hours).
- **HATEOAS (optional):** include `links` or `_links` with related resource URLs when API discoverability is a priority. At minimum, include `self` links on resources and `next`/`prev` on paginated collections.
- **Bulk operations:** support batch endpoints (`POST /users/batch`) for creating or updating multiple resources. Return individual results per item with status codes. Process items independently — one failure should not roll back others unless transactional semantics are required.
